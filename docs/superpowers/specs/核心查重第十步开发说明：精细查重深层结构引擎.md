# 核心查重第十步开发说明：精细查重深层结构引擎

## 1. 这一步解决什么问题

前面的“快速查重”已经能完成一轮可靠的结构筛查，但它更像“先快速找出可疑对象”。

它的核心优势是快，适合：

- 一个班级几十到上百份作业时先跑一遍
- 先把明显不相似的提交排除
- 给老师一个初筛结果

但它也有一个天然限制：

- 如果两份代码只是“节点袋子”很像，但真正的结构层次、方法组织方式、父子路径关系不一样，快速模式可能还会给出比较高的分数

所以这一步我们新增了第二套模式：

- `快速查重（FAST）`：继续保留原有 AC bag-of-nodes
- `精细查重（DEEP）`：在原有基础上，再增加更深层的结构特征比较

这样老师在前端发起查重时，就可以主动选择：

- 先快速筛查
- 或者直接用更细的结构模式做重点复核

结果仍然统一进入“查看结果”，不会拆成两套结果中心。

---

## 2. 这一步用了什么技术

这一步主要还是基于 **JavaParser + AST**，没有引入新的外部大模型或复杂中间件。

核心技术点有 4 个：

### 2.1 JavaParser AST

先把学生提交的 `.java` 文件解析成抽象语法树（AST）。

这样系统看到的就不是简单的文本，而是：

- 类
- 方法
- if / for / while
- return
- 表达式
- 方法调用

这些结构化节点。

### 2.2 规范化（Normalization）

为了降低“改名、改常量、改类型”带来的干扰，我们继续沿用原来的规范化策略：

- 变量名统一成 `ID`
- 方法名统一成 `ID`
- 类名统一成 `ID`
- 类型统一成 `TYPE`
- 字面量只保留种类，不保留具体值

这样系统更关注“结构怎么写”，而不是“名字怎么起”。

### 2.3 深层结构特征（Deep Signatures）

这一步不是只看普通 signature，而是新增了 3 层更细的结构特征：

#### （1）深度特征

同一种节点，如果出现在不同嵌套深度，语义可能不一样。

例如：

- 一个 `if` 在方法最外层
- 一个 `if` 在 `for` 里面
- 一个 `if` 在 `try-catch` 里面

我们会记录：

- `某类节点 + 深度桶`

这能更好地区分“看起来节点差不多，但嵌套层次不同”的代码。

#### （2）父子路径特征

只看节点本身还不够，还要看它“挂在哪个父节点下面”。

例如：

- `Return` 出现在 `If` 下面
- `Return` 出现在方法体直接下面

这两个结构感觉就不一样。

所以我们新增：

- `Path:父节点 -> 当前节点`

这种路径级 signature。

#### （3）方法级画像

一份提交通常不止一个方法。

快速模式是把整份提交揉成一个大特征袋；
精细模式会额外把每个方法单独做一份画像，再去比较“方法和方法”之间的相似度。

这样可以更好地区分：

- 两份代码只是全局元素相似
- 还是具体某些方法真的很像

### 2.4 加权融合评分

精细模式最后不是只看一个分数，而是组合三部分：

- 粗粒度结构分（coarseScore）
- 深层结构分（deepStructuralScore）
- 方法级匹配分（methodScore）

最后合成一个总分 `finalScore`。

当前实现采用：

- 粗粒度结构：`35%`
- 深层结构：`45%`
- 方法级匹配：`20%`

这样做的原因是：

- 保留快速模式的稳定性
- 同时让“嵌套层次、父子路径、方法组织”对最终分数产生更明显影响

## 3. 具体是怎么开发的

这一步开发不是直接改一大坨旧代码，而是分成 4 小步。

### 3.1 先补失败测试

我们先写了两组测试：

#### 测试一：深度提取器测试

验证：

- 是否真的提取了 `@depth=`
- 是否真的提取了 `Path:`
- 是否真的提取了方法级画像

#### 测试二：精细模式执行测试

验证：

- 发起 `DEEP` 模式时，是否真的进入精细引擎
- 结果 payload 里是否带有：
  - `plagiarismMode=DEEP`
  - `algoVersion=AC_DEEP_AST_V1`
  - `engineType=DEEP_STRUCTURE`
  - `methodMatches`
  - `scoreBreakdown`

这一步的目的很简单：

- 先把“我们想要的结果”写下来
- 再让代码去满足这些结果

这样开发更稳，不容易写偏。

### 3.2 新增深度结构提取器

新增了：

- `DeepAstSignatureExtractor`

它负责把一份 Java 源码提取成：

- 粗粒度 signature
- 深度 signature
- 路径 signature
- 方法级画像

最终形成：

- `DeepAstProfile`

### 3.3 新增深度相似度计算器

新增了：

- `DeepAstSimilarityCalculator`

它负责：

1. 先算粗粒度分
2. 再算深层结构分
3. 再算方法级最佳匹配分
4. 最后把三者融合成总分

同时输出：

- `coarseScore`
- `deepStructuralScore`
- `methodScore`
- `finalScore`
- `methodMatches`

### 3.4 接入现有查重任务执行链

在 `TeacherPlagiarismExecutionService` 中，我们没有推翻原有任务流程，而是做了模式分流：

- `FAST` → 继续走原有快速引擎
- `DEEP` → 走新的深层结构引擎

这样好处很大：

- 旧功能不受影响
- 前端只多一个模式选择
- 数据库结构不用重建
- 结果页也不用拆成两套

## 4. 这一版比快速查重“更强”在哪

可以把两种模式理解成：

### 快速查重

像是先看“零件清单”像不像。

如果两份代码都用了：

- if
- return
- call
- binary expr

那它会觉得挺像。

### 精细查重

像是在看：

- 零件清单
- 零件装配方式
- 嵌套深度
- 方法划分方式

所以它更能区分：

- “节点差不多，但结构组织不同”
- “真的是结构层次和方法逻辑都很接近”

这就是它明显强于快速模式的地方。

---

## 5. 结果页里新增了什么信息

为了让老师最终还能在同一个结果中心里看，我们把新增信息放进了证据 payload：

- `plagiarismMode`
- `engineType`
- `algoVersion`
- `scoreBreakdown`
- `methodMatches`

这样老师虽然仍然在同一个“查看结果”页面看结果，
但后台其实已经能分辨：

- 这条结果是快速模式跑出来的
- 还是精细模式跑出来的

同时以后也更方便继续升级。

---

## 6. 这一步的收益

### 对老师

- 有了“快速 / 精细”两个模式可选
- 结果仍在同一个地方看，不会乱
- 精细模式更适合重点复核

### 对系统

- 不破坏已有快速模式
- 新模式走独立分支，便于后续继续迭代
- 证据结构更丰富，后面更容易接“代码片段级对比”

### 对论文 / 答辩

这一步很适合作为一个亮点来讲：

- 系统不是只有一种查重算法
- 而是设计成了“双模式查重架构”
- 能兼顾速度与精度

---

## 7. 当前仍然有哪些边界

虽然这一步已经比快速模式明显更强，但它还不是最终形态。

当前边界主要有 3 个：

### 7.1 方法级匹配还是贪心匹配

现在方法级匹配使用的是“从左到右找最佳方法”的贪心方式。

优点：

- 实现简单
- 速度可控

缺点：

- 不一定是全局最优匹配

后面如果需要，可以升级为更严格的最优分配。

### 7.2 还没有做到代码片段级精准高亮

目前精细模式已经能提供更细的结构证据，
但“精确到哪几段代码命中”的真实片段定位，还需要继续做下一步增强。

### 7.3 权重还是经验值

当前的 `35% + 45% + 20%` 是工程上合理、可解释的初始配置，
后续还可以根据样本数据继续调优。

---

## 8. 下一步怎么继续做

这一步之后，最自然的下一步就是：

### 下一步：代码片段级命中定位

目标是让精细模式不仅能说“这两份代码结构很像”，
还可以进一步告诉老师：

- 哪个方法最像
- 哪几段代码最值得重点看
- 在前端 IDE 风格对比页里，把这些代码块直接高亮出来

这样老师看到的就不只是一个分数，
而是“为什么像、像在哪里”。
