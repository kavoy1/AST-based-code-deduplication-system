# 查重核心方案与性能优化建议（50到100份作业）

日期：2026-03-28  
适用范围：`D:\学校\毕业设计\项目\graduation_project` 当前 Java AST 查重后端实现与 `2026-03-16-ast-plagiarism-3role-ai-design.md` 中的主方案。

## 1. 写本文档的原因

当前系统的查重主线已经明确：以“班级 -> 作业”为单位，在截止后对该班级下的学生最新有效提交执行 AST 规则化与结构相似度计算。  
当一个班级的某次作业提交量达到 **50~100 份** 时，理论上的两两比较数量分别为：

- 50 份提交：`50 * 49 / 2 = 1225` 对
- 100 份提交：`100 * 99 / 2 = 4950` 对

如果继续使用“单线程 + 每次 Job 重新读文件 + 逐对落库 + 逐对更新进度”的方式，教师点击“一键查重”后的等待时间会明显变长，体验较差。  
因此，本文档的目标不是推翻原有方案，而是：**在保留当前查重判定口径不变的前提下，给出适合 50~100 份作业规模的工程优化方案。**

---

## 2. 现有解析查重核心方案总结

### 2.1 业务边界

当前系统的业务边界是：

- 以“班级 -> 作业”为单位执行查重
- 学生可提交单个或多个 `.java` 文件
- 一次提交可以包含多个 Java 文件
- 教师只能在截止后查看报告并触发查重
- AI 只负责解释增强，不参与最终判定

默认比较口径：

- 只使用 `is_latest = 1` 的最新提交版本
- 只使用 `is_valid = 1` 的有效提交
- 只在同一班级内进行两两比较，不跨班级比较

### 2.2 解析与归一化主线

当前主线采用：

1. 读取提交中的 `.java` 文件
2. 使用 `JavaParser` 解析为 AST
3. 对 AST 做全局归一化
4. 为节点生成 `signature`
5. 统计每份提交的 `signature -> count`
6. 基于 AC 公式执行两两比较
7. 为超过阈值的结果生成证据并落库

当前归一化规则的核心思想是：尽量削弱“改名、改常量、局部改写”这类表层变化的影响，突出结构层面的相似性。

### 2.3 当前归一化规则摘要

按设计文档，主要归一化规则如下：

- 所有标识符名统一替换为 `ID`
- 所有字面量不保留具体值，只保留类别
- 所有类型名统一替换为 `TYPE`
- 不关心注释、格式、空白符等文本层差异

当前 `signature` 规则的核心项包括：

- `BinaryExpr:<op>`
- `UnaryExpr:<op>`
- `Assign:<op>`
- `Literal:<kind>`
- `Call:arity=<n>`
- `If:else=<0|1>`
- `For:mask=<init,cond,update>`
- `ForEach`
- `While`
- `DoWhile`
- `Return / Break / Continue / Throw / TryCatch / Switch`

这套规则本质上是：**把代码提交转化为一个“节点签名多重集合（multiset）”。**

### 2.4 AC 相似度定义

当前系统采用的主判定指标是 AC：

- 两份提交的节点总数分别记为 `N1`、`N2`
- 两份提交中每种 `signature` 的计数分别记为 `count1[s]`、`count2[s]`
- 匹配节点数定义为：

`M = Σ min(count1[s], count2[s])`

- 最终相似度定义为：

`AC = 2 * M / (N1 + N2)`

性质：

- `AC` 取值范围是 `0 ~ 1`
- 越接近 `1`，说明结构成分越相似
- 这是典型的 **bag-of-nodes 结构成分相似度**
- 它对“改名、改常量值”较鲁棒
- 它对深层嵌套结构的整体重排不够敏感

### 2.5 多文件提交的聚合方式

当前口径不是“文件对文件匹配”，而是：

- 每个 `.java` 文件先独立提取 `signature -> count`
- 同一份提交内的多个文件直接相加，形成提交级大 multiset
- 最终按“提交 vs 提交”算一次 AC

这个设计的优点是：

- 实现简单
- 证据可解释性强
- 不需要文件级对齐或匈牙利匹配
- 对 50~100 份作业的工程复杂度更友好

### 2.6 宽松解析策略

当前采用“宽松”策略：

- 逐文件解析
- 解析成功的文件参与签名统计
- 解析失败的文件记录 `parse_error`
- 若 `parse_ok_files >= 1`，则该提交视为有效
- 若 `parse_ok_files = 0`，则该提交不参与两两比较，并进入“不可比对名单”

这个策略适合教学场景，因为学生提交中经常会混入一个错误文件，但不应导致整份提交直接失去比较资格。

### 2.7 证据与 AI 的角色

当前证据主线是：

- 证据类型：`SIGNATURE_OVERLAP_TOP`
- 输出 TopN 重叠贡献最大的 `signature`
- 同时输出 `N1 / N2 / M / AC`
- 附带解析失败文件摘要

AI 的定位是：

- 只对证据做解释增强
- 不参与最终判定
- 仅教师手动触发
- 默认不发送完整源码给大模型

## 3. 当前实现下的主要性能瓶颈

结合当前后端实现，50~100 份作业变慢的原因主要不是 AC 公式本身，而是工程执行方式。

### 3.1 每次 Job 都重新读取并重新解析源码

当前执行链路中，查重任务启动后会：

- 查询所有有效提交
- 查询每份提交的文件列表
- 从磁盘重新读取每个文件
- 对每个文件重新解析 AST
- 重新做归一化与签名提取

这意味着：

- 同一份提交如果教师反复点“重新查重”，会重复做同一套文件 IO 与 AST 解析
- 真正消耗时间的不是 AC 公式，而是“读文件 + 解析 AST + 归一化”

对于 50~100 份作业，这部分通常比“最终相似度计算”更耗时。

### 3.2 存在 N+1 查询与重复 IO

当前实现按提交逐个查询 `submission_file`，再逐个读文件，容易形成：

- 提交列表 1 次查询
- 每个提交再查一次文件列表
- 每个文件再做一次磁盘读取

当提交数量提升到 100 份时，这种“逐提交、逐文件”的访问方式会放大数据库往返和文件系统开销。

### 3.3 逐对更新 Job 进度，数据库写压力偏高

当前逻辑每比较完一对提交，就会立即：

- `progressDone + 1`
- `update plagiarism_job`

如果一个班级有 100 份提交，则需要执行 `4950` 次进度更新。  
这类高频小写操作对数据库和事务日志都不友好，而且本身不创造业务价值。

### 3.4 相似对与证据逐条插入

当前是：

- 命中阈值就插入一条 `similarity_pair`
- 然后再插入一条 `similarity_evidence`

如果命中结果较多，数据库会出现大量细粒度插入，吞吐偏低。

### 3.5 当前是“精确全量两两比较”，没有候选召回层

对于 50~100 份提交，`1225 ~ 4950` 对比较在理论上仍可接受。  
但如果每一对都在完整的特征空间上精确计算，再叠加文件重复解析、频繁写库，整体等待时间就会变大。

### 3.6 单线程同步执行，教师等待感明显

当前方案更像“教师点按钮 -> 后端立即执行完整查重”。  
如果班级作业规模达到 80~100 份，教师会明显感受到页面等待或任务处理时间较长。

## 4. 优化原则

针对 50~100 份作业的规模，我建议遵守下面四条原则：

### 原则 1：不先改判定口径，先改工程执行方式

当前 AC 方案在教学场景下已经够清晰、够稳定。  
问题主要在执行链路，而不是公式本身。  
因此第一步不建议换算法，而建议优化：

- 解析时机
- 缓存策略
- 批处理策略
- 异步执行策略

### 原则 2：50~100 份规模不必急着上复杂近似算法

100 份提交只有 `4950` 对。这个数量不算大。  
对于本科毕设来说，优先把精确方案做稳，比过早引入复杂召回算法更重要。

也就是说：

- **全量精确比较仍然可保留**
- 但要把“每次重复解析源码”和“高频写库”去掉

### 原则 3：把“重复成本”前移或缓存

提交一旦完成，后续重复查重时尽量不要再重复做：

- 读源文件
- 解析 AST
- 归一化
- 构建签名统计

这部分应该缓存到提交级别，而不是在每次 Job 中重新计算。

### 原则 4：让教师看到“任务在跑”，而不是一直等

即使总耗时没有缩短到秒级，只要：

- 异步执行
- 有进度
- 有状态
- 可稍后查看报告

教师端的体验就会明显好很多。

## 5. 面向 50~100 份作业的优化方案

下面按“优先级”给出建议。前四项建议优先落地。

### 5.1 优先级 A：提交时预提取并缓存签名画像

#### 方案

在学生提交成功后，立即对提交中的有效 `.java` 文件做一次：

- AST 解析
- 归一化
- `signature -> count` 提取
- 统计总节点数 `N`

然后把结果保存为“提交画像（Submission Profile）”。

可落地为：

- 数据库存 `submission_profile_json`
- 或单独建表 `submission_profile`
- 或落本地 JSON 文件并在表中保存路径

#### 收益

这样做之后，教师发起查重 Job 时：

- 不需要重新读每个源码文件
- 不需要重新解析 AST
- 只需要直接读取提交级画像
- 两两比较时只做 `Map<String, Integer>` 之间的计算

#### 结论

这是 **最重要、最值得优先做** 的优化。

### 5.2 优先级 A：把查重任务改成真正的异步 Job

#### 方案

教师点击“一键查重”后：

1. 先创建 `plagiarism_job`
2. 状态置为 `QUEUED`
3. 后台线程池或任务队列异步执行
4. 前端轮询任务状态与进度

#### 收益

- 教师不必同步等待
- 页面体验更稳定
- 后续更方便加“取消任务”“重试任务”“历史任务”

#### 推荐实现

对于毕设规模，不必引入 MQ。使用以下方式已经足够：

- Spring `@Async`
- 固定大小线程池
- 作业级串行、全局限并发

### 5.3 优先级 A：降低 Job 进度写库频率

#### 方案

不要每比较一对就更新一次数据库。改为：

- 每完成 20 对、50 对，或
- 每隔 1 秒

再批量更新一次进度。

#### 收益

在 100 份提交时，可把 `4950` 次数据库更新降到几十次，效果非常明显。

#### 推荐口径

- `progressTotal`：该班级有效提交的总 pair 数
- `progressDone`：已完成 pair 数
- 更新节流：`每 50 对或每 1 秒更新一次`

### 5.4 优先级 A：批量查询与批量写入

#### 方案

读数据时：

- 一次查出某个作业下的全部有效提交
- 一次查出这些提交对应的全部 `submission_file`
- 在内存中按 `submissionId` 分组

写数据时：

- 相似对先在内存缓冲
- 证据先在内存缓冲
- 达到一定批次后批量插入

#### 收益

- 降低数据库往返次数
- 降低事务碎片化
- 对 100 份提交规模会明显更稳

### 5.5 优先级 B：按班级分组后并行计算

#### 方案

当前查重本身就以班级为边界，可以按班级自然拆分：

- 班级 A 的 pair 集合一组任务
- 班级 B 的 pair 集合一组任务

同一个班级内部如果人数较多，还可以进一步做分片并行。

#### 推荐程度

对于单班 50~100 份提交，建议先做 **2~4 线程的有界并行**，不要盲目开大线程数。

#### 原因

AC 计算本身不重，真正开销常常在：

- Map 访问
n- JSON 组装
- 数据库存储

线程过多反而会让数据库和 GC 压力变大。

### 5.6 优先级 B：加入轻量级候选过滤

如果后续想进一步加速，但又不想上复杂的近似召回，可以先加非常简单的候选过滤条件。

#### 可用过滤条件

在进入精确 AC 计算前，先排除明显不可能高相似的 pair：

- **节点规模差过滤**：若 `min(N1, N2) / max(N1, N2)` 太低，则直接跳过
- **文件数差过滤**：若两个提交文件数差异过大，则降低优先级或跳过
- **粗签名重叠过滤**：先比较少量粗粒度特征，再决定是否进入完整 AC

#### 建议

对于 50~100 份规模，这一步不是必须项，但可以作为可选优化。

### 5.7 优先级 C：引入近似召回，只对候选做精确 AC

如果未来规模上升到：

- 单班 200 份以上，或
- 多班并行查重很多，或
- 教师频繁重复运行 Job

那么可以进一步引入候选召回层，例如：

- MinHash / LSH
- SimHash
- 粗粒度签名向量的倒排索引

执行流程改为：

1. 先做低成本候选召回
2. 只对候选 pair 做完整 AC
3. 对高分 pair 生成证据并落库

这一步更适合后续扩展，不建议作为当前毕设第一优先级。

### 5.8 优先级 C：提交级画像持久化版本化

如果教师可能多次重跑同一作业，建议把提交画像做成可复用资产：

- 跟随 `submissionId` 固定保存
- 如果提交内容不变，则画像不变
- 多次 Job 直接复用

这样后续还能支持：

- 重新调整阈值后快速重跑
- 调整 TopK 后快速重算视图
- 导出历史报告时复用已有画像

## 6. 推荐的分阶段落地路线

### 阶段一：先解决“明显慢”的问题

优先做：

1. 提交时生成并缓存提交画像
2. 查重任务改为异步执行
3. 进度更新节流
4. 批量查询、批量写入

这四项做完后，50~100 份作业的处理速度和教师体验会有明显改善。

### 阶段二：提升并发与稳定性

再做：

1. 按班级或分片并行计算
2. 增加历史 Job 列表
3. 增加任务失败重试与错误日志
4. 增加任务监控字段（耗时、画像复用率、命中 pair 数）

### 阶段三：为更大规模预留扩展

后续如人数继续增大，再考虑：

1. 候选召回层
2. 签名字典化与整数 ID 化
3. 更细的缓存与索引优化
4. 更复杂的报告统计视图

## 7. 对当前项目的明确建议

结合你当前毕设阶段，我建议采用下面这个结论：

### 结论 1：不要更换 AC 主方案

当前 AC 公式与 `bag-of-nodes` 主方案可以继续保留。  
对于本科毕设与 50~100 份作业规模，它是足够清晰、足够可解释、也足够稳的。

### 结论 2：先把“重复解析源码”改成“提交后一次提取，Job 中直接复用”

这是当前最关键的优化点。  
如果不做这一步，后续不管是并行还是节流，收益都有限。

### 结论 3：100 份以内不必急着引入复杂近似算法

4950 对全量精确比较本身不是不可接受，真正拖慢系统的是工程实现细节。  
所以现阶段更适合做“工程优化”，而不是“算法替换”。

### 结论 4：教师端必须看到异步任务与进度

即使总耗时没有降到极短，只要教师点击后：

- 任务进入队列
- 有状态变化
- 有进度条
- 可稍后回来查看

整体体验就已经从“卡住了”变成“系统在工作”。

---

## 8. 一句话总结

**当前查重核心方案不需要推翻；对 50~100 份作业，最有效的优化不是更换 AC，而是把“重复解析源码”改成“提交级画像复用”，再配合异步 Job、进度节流、批量查询与批量写入。**
