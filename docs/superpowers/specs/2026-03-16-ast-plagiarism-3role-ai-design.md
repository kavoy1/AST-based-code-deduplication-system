# 基于 Java 的代码作业自动查重系统（三端 + AST + QWEN）设计说明

日期：2026-03-16  
范围：以“某门课/某个班级的某次作业”为单位；学生提交截止后，教师可查看查重报告；学生仅可查看与自身相关的有限结果（可配置）。

## 1. 背景与目标

### 1.1 背景
编程课程中存在“改名换变量/改常量/改顺序”等抄袭方式。基于文本/字符串相似度的检测易被规避。本系统以 Java 代码为输入，通过 AST（抽象语法树）解析与结构相似度度量实现更鲁棒的查重，并提供三端（管理员/教师/学生）完整业务闭环。

### 1.2 目标（Must）
- 支持教师按“班级 -> 作业”发布与收作业，学生按作业上传单个/多个 `.java` 文件提交。
- 截止时间后，教师可对某次作业一键发起查重，生成可导出的报告与可解释证据。
- 查重判定以 AST 规则化 + 结构相似度为主；AI（QWEN）用于“解释增强”，不作为唯一判定依据。
- 权限隔离：学生不可查看其他学生的代码/详细相似对；教师只可管理自己班级；管理员可全局治理。

### 1.3 非目标（Non-goals，阶段 1 明确不做）
- 不做全平台历史全量查重（可作为阶段 2 扩展）。
- 不做项目压缩包（Maven/Gradle）上传与构建；仅 `.java` 文件集合。
- 不承诺 100% 准确率；提供“人工复核工作流”与“误报标记”能力。

## 2. 角色与功能范围

### 2.1 学生端
- 班级：通过邀请码申请加入；查看班级公告、作业列表。
- 作业提交：
  - 查看作业详情、截止时间、提交状态。
  - 上传单个/多个 `.java` 文件；支持多次提交（策略可配置：保留最新/保留版本）。
  - 截止前可查看基础校验状态（如：文件数量限制、可选的语法解析成功/失败）。
- 结果查看（截止后）：
  - 仅展示与本人相关的摘要：是否被标记、相似度区间、教师备注/通知（避免互相抄作业“对答案”）。
- 个人中心：资料维护、密码修改、通知、反馈。

### 2.2 教师端
- 班级管理：创建班级、生成邀请码、审核入班申请、学生名单管理（移除/禁用入班）。
- 作业管理：
  - 发布作业：标题、描述、截止时间、允许文件数、是否允许多次提交、截止前是否允许撤回（可选）。
  - 查看提交率、迟交统计（迟交策略可选：不允许/允许但标记）。
- 查重与报告：
  - 截止后“一键查重”：可设置阈值、TopK、是否排除白名单（如同组协作名单/教师确认的模板代码）。
  - 报告内容：相似对列表、相似度、证据摘要、风险等级、处理状态（待确认/已确认/误报）。
  - 导出：Excel/PDF（至少一种）。
- 处置闭环：
  - 对相似对标记状态、备注、发通知（站内信/公告/可选邮件）。

### 2.3 管理员端
- 用户与角色：账号管理、禁用/重置、教师注册邀请码策略。
- 全局配置：上传大小与数量限制、存储路径、查重阈值默认值、查重并发、日志与审计。
- 公告与反馈：系统公告、反馈工单处理。
- AI 配置：QWEN 接入参数（API Key/地址/模型名）、额度/频控、脱敏与失败降级策略。

## 3. 核心业务流程（以作业为单位）

### 3.1 发布与提交
1) 教师创建班级，生成邀请码。  
2) 学生申请加入班级，教师审核通过。  
3) 教师发布作业（设置截止时间与提交规则）。  
4) 学生上传 `.java` 文件集合提交（可多次提交）。  

### 3.2 截止后查重与处置
1) 达到截止时间后，教师在作业详情页触发“一键查重”。  
2) 后端创建查重任务（Job），进入队列异步执行：解析 -> 规范化 -> 相似度计算 -> 生成报告。  
3) 教师查看报告：Top 相似对、证据、处理状态。  
4) 教师对相似对进行确认/误报标记，导出报告并通知相关学生。  

## 4. 查重技术方案（主线 A + 解释增强 C）

### 4.1 输入与基本约束
- 输入：某次作业下的所有学生“提交版本”（默认仅最新一次；或按策略选择）。
- 每个提交由多个 `.java` 文件构成。
- 解析器：JavaParser（或等价 AST 工具）解析为 AST。

### 4.2 AST 规范化（Normalization）
目标：降低“改名/改常量/换顺序”的干扰，提高结构一致性。
- 标识符归一化：变量/方法/类/参数/字段等标识符名统一替换为 `ID`。
- 常量归一化：数字/字符串常量映射为类型占位符（如 `NUM`, `STR`）或区间桶。
- 类型名归一化：所有类型名统一替换为 `TYPE`（不细分）。
- 结构规整化：
  - `for/while/do-while` 统一抽象为“循环节点”并保留关键子结构（初始化/条件/更新的存在性）。
  - `if/else if/else` 统一为条件链结构。
  - 可选：对无关顺序的成员（import、字段、方法）按稳定键排序（需谨慎，避免破坏语义过多）。
- 去噪：注释与格式信息不进入 AST；可选过滤空语句、冗余括号节点等。

输出：每份提交得到一个或多个“规范化 AST（按文件/按类/按方法）”以及可追溯的映射信息（用于证据回放）。

### 4.3 结构相似度（AC）计算（Bag-of-Nodes）
本项目主相似度指标采用“节点签名多重集合（multiset）交集占比”的 **AC**（0~1）：

- 记两棵（归一化后的）AST 节点总数为 `N1`、`N2`
- 记两棵 AST 的“可匹配节点数”为 `M`
- 定义：`AC = 2*M / (N1 + N2)`

其中 `M` 的计算方式为：
- 对两棵 AST 分别统计每一种 `signature` 的出现次数 `count1[signature]`、`count2[signature]`
- `M = Σ_signature min(count1[signature], count2[signature])`

> 备注：这是一种“bag-of-nodes 结构成分相似度”，对整体嵌套形状不敏感；对改名/改常量值的鲁棒性较强，但对深层结构重排的敏感性较弱（阶段 1 可接受）。

#### 4.3.1 多文件提交聚合口径
一次提交（Submission）可能包含多个 `.java` 文件。聚合口径为：
- 逐文件解析并提取 `signature -> count`
- 将同一提交内所有文件的 `signature` 计数直接相加，形成“提交级别的大 multiset”
- 两份提交之间的 AC 按上述公式计算一次（不做文件级对齐/匈牙利匹配）

#### 4.3.2 signature 最小关键属性清单（V1）
对每个 AST 节点生成 `signature`。V1 最小关键属性清单如下：
- `BinaryExpr`：`BinaryExpr:<op>`（必须包含运算符：`+ - * / %`，`< <= > >= == !=`，`&& ||`，以及 Java 支持的位运算）
- `UnaryExpr`：`UnaryExpr:<op>`（如 `! - ~ ++ --`）
- `Assign`：`Assign:<op>`（如 `= += -= *= /= ...`）
- `Literal`：`Literal:<kind>`（`NUM/STR/BOOL/CHAR/NULL`）
- `Call/MethodCall`：不保留调用名；`Call:arity=<n>`（可选再加 `hasReceiver=0/1`）
- `If`：`If:else=<0|1>`（不看条件细节）
- `For`：
  - C/Java `for`：`For:mask=<init,cond,update>`（三段是否存在，如 `101`）
  - `foreach`：用不同节点类型名（如 `ForEach`）
- `While/DoWhile`：只看节点类型（不看条件细节）
- `Return/Break/Continue/Throw/TryCatch/Switch`：默认只看节点类型  
  - 可选增强（后续版本）：`TryCatch:finally=<0|1>,catchCountBucket`；`Switch:hasDefault=<0|1>,caseCountBucket`

#### 4.3.3 阈值与 TopK（阶段 1 默认）
- 查重阈值：`threshold = 0.80`（落库 `score=round(AC*100)`，阈值对应 `score>=80`）
- 报告视图：每个学生 TopK：`TopK = 10`
- Job 策略：允许同一作业（assignment）反复创建 Job 并保留历史（报告默认展示该作业“最新一次 DONE Job”）

#### 4.3.4 解析失败（宽松策略）
- 逐文件解析；成功的文件参与签名累加；失败的文件记录 `parse_error`（文件名 + 原因摘要），不纳入签名
- `submission.is_valid` 定义为：**本次 Job 中** `parse_ok_files >= 1`
- 若 `parse_ok_files = 0`：该 submission 不参与两两计算；报告页单独输出“解析失败/不可比对名单”（含失败原因摘要）

### 4.4 证据生成（Explainability）
必须能回答“为什么相似”：
- 阶段 1（与 AC 配套）：基于 signature overlap 生成证据（不做树对齐）。
- 输出 TopN：贡献最大的 signature 列表（`matchedCount=min(c1,c2)`），并给出总计（`N1,N2,M,AC`）。
- 解析失败摘要：如果任一侧存在解析失败文件，给出文件名与原因摘要（不暴露代码全文）。

#### 4.4.1 Evidence payloadJson（稳定 V1）
为保证前端/历史数据兼容，证据结构采用稳定的 V1 Schema（`schemaVersion=1`）：
- `type`: `SIGNATURE_OVERLAP_TOP`
- `summary`: 面向教师的 1 段话（Top 命中、覆盖率/比例、是否存在解析失败文件）
- `payloadJson`（建议字段）：
  - `schemaVersion: 1`
  - `totals: { N1, N2, M, AC }`
  - `topN: [{ signatureId, signature, countA, countB, matchedCount, contribution, contributionRatio }]`
  - `parseFailures: { sideA: [{file, reason}], sideB: [{file, reason}] }`
- `weight`: 证据粗权重（建议用 `M`，便于排序）

其中 `topN[i].signature` 落库采用“两段式”：
- `signatureId`：对规范化后的 signature 文本做稳定短 Hash（用于去重/前端 key/排序稳定）
- `signature`：可读的原始 signature 文本（便于导出与排障）

## 5. QWEN（AI）接入设计（解释增强，不做判定）

### 5.1 触发点
- 教师打开某个“相似对”详情页时，点击“生成 AI 解释”（或自动生成但需可开关）。

### 5.2 输入与脱敏
输入优先使用“规则查重输出的证据结构”而非提交代码全文：
- 相似度分数、证据子树摘要、热点方法/类的匿名标识、差异统计（重命名比例/常量替换比例/结构改动比例）。
脱敏策略（管理员可配置）：
- 默认不发送学生姓名/学号/邮箱。
- 默认不发送完整源代码；必要时仅发送极短片段（并可开关）。

### 5.3 输出与落库
输出结构：
- 中文解释（面向教师）：相似点概述、可疑改动类型、建议核查点。
- 风险等级建议（低/中/高）与建议操作（复核/面谈/忽略/加入白名单）。
落库：与相似对绑定，记录生成时间、模型、耗时、状态（成功/失败/降级）。

### 5.4 失败降级
- 超时/限流/异常：展示“规则解释”版本（不阻断教师查看报告）。
- 频控：按教师/作业维度限次；避免一键生成导致成本不可控。

## 6. 数据模型（概念层）

> 注：具体表结构可在实现计划阶段对齐现有库表/实体。

- `Clazz`：班级（teacherId、inviteCode、创建时间…）
- `Assignment`：作业（clazzId、title、description、deadline、规则配置…）
- `Submission`：提交（assignmentId、studentId、submitTime、版本号/是否最新、状态…）
- `SubmissionFile`：提交文件（submissionId、filename、存储路径/内容哈希、解析状态…）
- `PlagiarismJob`：查重任务（assignmentId、createTime、status、params、耗时…）
- `SimilarityPair`：相似对（jobId、studentA、studentB、score、status、teacherNote…）
- `SimilarityEvidence`：证据（pairId、type、summary、weight、payloadJson…）
- `AiExplanation`：AI 解释（pairId、model、promptHash、result、status、latency…）
- `Whitelist`：白名单（assignmentId 或 clazzId；可包含模板哈希/文件哈希/方法特征…）

### 6.1 迟交与版本口径（阶段 1 默认）
- 截止前：允许提交
- 截止后：允许提交但标记迟交（`is_late=1`，并记录 `submit_time` 与 `deadline_at`）
- 若 `allow_resubmit=1`：同一学生允许多次提交；新提交置为 `is_latest=1`，旧版本保留历史（`is_latest=0`）
- 若 `allow_resubmit=0`：截止前提交过一次则拒绝后续提交（返回明确错误码/提示）
- 查重/评分默认只使用 `is_latest=1` 的版本；历史仅用于追溯

## 7. 权限矩阵（简化）
- 学生：
  - 可提交自己的作业；可查看自己的提交历史与自身摘要结果（截止后）。
  - 不可查看他人提交/相似对详情/证据。
- 教师：
  - 仅可管理自己班级及其作业/提交/报告。
  - 可查看相似对证据、AI 解释、导出报告、发通知。
- 管理员：
  - 全局用户与配置；可查看系统级日志与统计；不参与教学业务判定（可只做治理）。

## 8. 接口范围（高层）
后端接口需覆盖：
- 班级：创建/列表/邀请码/申请审核/学生名单
- 作业：发布/修改/详情/列表
- 提交：上传 `.java` 文件（multipart）、提交记录、下载
- 查重：创建 Job、查询 Job 状态、获取报告列表与相似对详情、标记相似对状态、导出
- AI：为相似对生成/获取解释、配置读取（管理员）

### 8.1 关键权限点（阶段 1）
- 教师：只能访问自己班级/作业；且只能在截止后创建查重 Job
- 学生：只能提交/查看自己的作业提交；查重结果摘要需教师已创建并完成 Job 后才可见（默认 A）
- 管理员：全局治理与配置；不参与教学判定

前端页面需覆盖：
- 学生：班级/作业列表/提交页/个人中心/通知
- 教师：班级管理/作业管理/提交管理/查重报告/相似对详情
- 管理员：用户管理/公告/反馈/系统配置/AI 配置

## 9. 验收标准（阶段 1）
- 能完成一个班级内一次作业的：发布 -> 提交 -> 截止 -> 查重 -> 生成报告 -> 教师标记与导出。
- 查重报告至少包含：相似对 TopN、相似度分数、证据摘要；支持阈值过滤。
- QWEN 解释可按相似对生成并展示；失败时可降级到规则解释。
- 权限验证：学生无法获取他人代码/相似对；教师无法越权访问其他教师班级数据。

## 10. 风险与对策
- 误报/漏报：通过白名单（模板/IO）、阈值可调、人工复核流程缓解。
- 性能：候选召回 + 分批计算；作业维度的任务队列与进度展示。
- AI 合规与成本：默认脱敏、限流、失败降级、可关闭自动生成。
