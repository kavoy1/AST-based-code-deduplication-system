# 核心查重第一步开发说明：提交级 AST 签名画像缓存

日期：2026-03-28  
适用项目：`D:\学校\毕业设计\项目\graduation_project`

## 1. 这一步要解决什么问题

在当前查重流程里，教师每点一次“一键查重”，后端都会重新做一遍下面这些事情：

1. 从数据库查出提交记录
2. 找到每份提交对应的 `.java` 文件
3. 从磁盘重新读取源码文件
4. 重新解析 AST
5. 重新归一化
6. 重新统计 `signature -> count`
7. 再进入两两比较

这套流程在功能上没有问题，但在性能上有一个明显问题：

- **同一份提交的 AST 画像会被重复计算很多次**

这就像我们已经把一篇作文批改过一次，但每次排名时还要重新通读全文、重新做笔记，浪费时间。

所以第一步优化不改 AC 算法，而是先做一件更划算的事：

> **学生提交代码时，就把这份提交的 AST 签名画像提前算出来并保存好。**

后面教师点查重时，直接取这个画像来比较，不再重新解析整份源码。

---

## 2. 什么是“提交画像”

这里的“提交画像”不是图像，而是这份提交在查重算法中的“特征快照”。

它至少包含：

- 这份提交总共有多少个有效 AST 节点：`totalNodes`
- 每种 `signature` 出现了多少次：`signatureCountsJson`
- 使用的是哪一版算法：`algoVersion`

可以理解为：

- 原始代码是“原材料”
- AST 是“结构化后的代码树”
- 提交画像是“最终用于比较的统计结果”

查重时真正需要的，其实不是源代码全文，而是这份画像。

---

## 3. 为什么这一步能明显提速

### 3.1 原来的慢点

原来每次 Job 都要重复做：

- 文件 IO
- JavaParser 解析
- 归一化
- 签名统计

这些步骤比最终的 AC 公式本身更耗时。

### 3.2 现在的优化思路

改成：

- 学生提交时做一次画像提取
- 结果直接入库
- 教师查重时直接取画像比较

这样后面重复查重时，只需要：

- 读取画像 JSON
- 转成 `Map<String, Integer>`
- 做 AC 比较

这个过程会比“重新读源码 + 重新解析 AST”快很多。

---

## 4. 这一步具体怎么开发

### 4.1 新增一张表：`submission_profile`

我们需要给每份提交单独保存一条画像记录。

建议字段如下：

- `id`：主键
- `submission_id`：关联到哪一份提交
- `algo_version`：算法版本，例如 `AC_BAG_OF_NODES_V1`
- `total_nodes`：这份提交的总 AST 节点数
- `signature_counts_json`：签名计数 JSON
- `parse_failures_json`：解析失败文件摘要 JSON
- `create_time`
- `update_time`

### 4.2 学生提交时生成画像

原本学生提交流程已经会做：

- 文件上传
- 语法解析
- 保存 `submission_file`
- 更新 `parse_ok_files`

这一步我们在它后面追加：

- 对解析成功的 `.java` 文件调用 `JavaAstSignatureExtractor`
- 把每个文件的签名统计合并到提交级别
- 写入 `submission_profile`

### 4.3 教师查重时优先复用画像

原本 `TeacherPlagiarismExecutionService` 是：

- 先查提交
- 再查文件
- 再读磁盘
- 再提取签名

这一步改为：

1. 先查 `submission_profile`
2. 如果这份提交已经有画像，就直接使用
3. 如果这份提交没有画像（比如旧数据），再回退到旧逻辑动态构建

这样做的好处是：

- 不会破坏旧数据
- 新提交自动享受优化
- 旧提交也还能继续查重

---

## 5. 这一步用了什么技术

这里给你按新手能理解的方式解释。

### 5.1 JavaParser

作用：把 Java 源码解析成 AST。  
你可以把它理解成“把代码翻译成程序能理解的树形结构”。

### 5.2 JSON 持久化

我们把 `signature -> count` 这种 `Map` 存成 JSON 文本。  
原因是：

- 结构灵活
- 开发简单
- 不需要把每种 signature 拆成很多张明细表

### 5.3 MyBatis-Plus

用它来做：

- `submission_profile` 的增删改查
- 关联 `submissionId` 读取画像

### 5.4 回退机制

这是工程上很常见的一种设计：

- 有新数据就走新方案
- 没有新数据就回到旧方案

这样系统不会因为升级而把旧数据“卡死”。

---

## 6. 为什么不直接把所有东西都改掉

因为核心查重已经能跑。现在最重要的是：

- 一步一步优化
- 每一步都可验证
- 每一步都不把已有功能搞坏

所以这一步只解决“重复解析源码”这个最大问题，不同时去改：

- 并发模型
- 候选召回
- 更复杂的索引结构
- 更复杂的缓存系统

这就是工程上常说的：

> **先解决最大瓶颈，再做第二层优化。**

---

## 7. 这一步完成后的效果

完成后，系统会变成：

- 学生提交时：生成并保存 AST 提交画像
- 教师查重时：优先读取画像做比较
- 旧数据：仍可回退到文件级解析逻辑

这样一来，后续继续做：

- 异步 Job
- 进度节流
- 批量写库
- 并行计算

都会更顺，因为“最重的重复工作”已经被提前消掉了。

---

## 8. 一句话总结

**这一步不是改查重公式，而是把“每次查重都重复解析源码”改成“提交时一次提取，后续重复复用”，这是当前最值得优先做的性能优化。**
