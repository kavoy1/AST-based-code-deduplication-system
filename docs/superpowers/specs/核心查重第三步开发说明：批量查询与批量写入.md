# 核心查重第三步开发说明：批量查询与批量写入

日期：2026-03-29  
适用项目：`D:\学校\毕业设计\项目\graduation_project`

## 1. 这一步要解决什么问题

前两步我们已经做了两件关键优化：

1. 提交时缓存 AST 签名画像
2. 查重任务异步化，并对进度更新做节流

但当前查重执行里还有两个明显的数据库热点：

- 旧数据没有画像时，仍然会按提交一条一条去查 `submission_file`
- 命中阈值的相似对和证据，仍然是一条一条插入数据库

这一步的目标就是把这两个热点收掉。

---

## 2. 什么叫“批量查询”

### 2.1 白话解释

假设一个班有很多提交，如果我们要查每份提交的文件列表，有两种方式：

- 方式 A：每份提交查一次
- 方式 B：一次把所有提交对应的文件都查出来，再在内存里分组

方式 A 的问题是：

- 提交越多，数据库往返次数越多
- 每次查询本身都不大，但来回很多次就会慢

方式 B 的核心思想是：

- **少跑几次数据库，多在内存里自己整理**

这就是“批量查询”。

### 2.2 这一步里怎么用

在查重执行中，如果某些旧提交还没有 `submission_profile`，就需要回退到文件级解析。  
这时候我们不再对每个提交单独查 `submission_file`，而是：

1. 找出所有缺少画像的 `submissionId`
2. 一次查出这些 `submissionId` 对应的全部文件记录
3. 在 Java 内存中按 `submissionId` 分组
4. 再逐份提交构建画像

---

## 3. 什么叫“批量写入”

### 3.1 白话解释

当前查重执行中，只要一对提交超过阈值，就会做：

- 插入一条 `similarity_pair`
- 插入一条 `similarity_evidence`

如果命中的 pair 比较多，这就会变成很多次小写操作。  
数据库并不喜欢这种“非常碎”的写入方式。

批量写入的思路是：

- 先把一批结果放到内存里
- 攒到一定数量再一起写入数据库

这就像：

- 不是每买一支笔就去结一次账
- 而是先把购物篮装一些，再统一结账

### 3.2 这一步里怎么用

我们会把命中的相似对先放进一个列表，比如凑够 100 条：

1. 批量插入 `similarity_pair`
2. 再根据这些 pair 的主键，组装对应的 `similarity_evidence`
3. 批量插入 `similarity_evidence`

这样数据库的写入压力会明显下降。

---

## 4. 这一步具体怎么开发

### 4.1 为 Mapper 增加批量方法

需要增加几个 Mapper 方法：

- `SubmissionFileMapper.selectBySubmissionIds(...)`
- `SimilarityPairMapper.batchInsert(...)`
- `SimilarityEvidenceMapper.batchInsert(...)`

其中：

- 第一个负责批量读文件记录
- 后两个负责批量写相似对和证据

### 4.2 在 MyBatis XML 中写批量 SQL

因为批量 SQL 用 `foreach` 更方便，所以会增加对应的 XML：

- `SubmissionFileMapper.xml`
- `SimilarityPairMapper.xml`
- `SimilarityEvidenceMapper.xml`

典型形式是：

- `where submission_id in (...)`
- `insert into ... values (...), (...), (...)`

### 4.3 执行器中的改法

`TeacherPlagiarismExecutionService` 里主要做两件事：

#### 改法一：旧数据回退时批量查文件

原来：

- 一个 `submissionId` 查一次文件列表

现在：

- 一次把全部缺少画像的提交文件查出来
- 按 `submissionId` 分组后再逐个处理

#### 改法二：命中结果先缓冲，再分批落库

原来：

- 每命中一对就立即写 pair 和 evidence

现在：

- 命中结果先放进缓冲区
- 达到批大小时统一写库
- 任务结束时再把剩余未写入的结果一次冲刷

---

## 5. 这一步用了什么技术

### 5.1 MyBatis XML 动态 SQL

作用：方便写：

- `IN (...)` 查询
- 多值 `INSERT`

这是 MyBatis 做批量 SQL 最常见的方式之一。

### 5.2 内存分组

把数据库查出来的多条 `submission_file` 记录，在 Java 里按 `submissionId` 分组。  
这样后续查重逻辑仍然能按“提交”来处理，不需要改算法结构。

### 5.3 缓冲区思想

命中结果不会马上写库，而是先放到内存列表里。  
这是一种典型的“批处理缓冲”做法。

---

## 6. 为什么这一步值得做

因为查重过程本身就是“很多小操作重复执行”。  
如果不做批处理，就会出现：

- 数据库查询次数太多
- 数据库插入次数太多
- 整体吞吐下降

而这一步做完后：

- 对旧数据的回退成本更低
- 高命中场景下写库更稳
- 为后续更大班级规模打下基础

---

## 7. 这一步不解决什么

为了控制风险，这一步不处理：

- 更复杂的召回算法
- 更复杂的多线程并行比对
- 更复杂的缓存淘汰机制
- 更复杂的报告聚合统计

也就是说，它仍然是“工程执行优化”，不是“换算法”。

---

## 8. 一句话总结

**这一步的本质，是把“多次零散地读数据库、写数据库”改成“少量集中地读、成批地写”，从而进一步缩短查重任务在 50~100 份作业场景下的执行时间。**
