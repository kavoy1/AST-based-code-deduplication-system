# 核心查重第七步开发说明：统计展示与教师端报告对接
日期：2026-03-29  
适用项目：`D:\学校\毕业设计\项目\graduation_project`

## 1. 这一步要解决什么问题？
前面第六步已经把查重任务的运行统计写进了 `plagiarism_job.config_snapshot`。  
也就是说，后端其实已经知道：
- 本次有多少可比对提交
- 理论上一共有多少 pair
- 因为节点规模上界被跳过了多少 pair
- 因为粗粒度桶上界被跳过了多少 pair
- 真正进入完整 AC 计算的有多少
- 最终超过阈值的有多少
- 这次任务是不是复用了旧结果

但是现在还有一个明显的断层：

> **数据已经有了，但老师在页面上还看不到。**

如果统计只留在后端快照里：
- 老师前端感知不到优化效果
- 开发者排查性能时还得去翻原始 JSON
- 你答辩时也不方便直接展示

所以这一步的目标是：

> **把已经存在的任务统计，正式整理成后端接口字段，并接到教师端页面里。**

---

## 2. 这一步为什么不只是“前端改一下”？
因为这不是简单把几行字画到页面上。  
要真正做好，需要把一整条链路打通：

1. 后端把 `config_snapshot` 里的统计解析成稳定字段
2. API 文档明确这些字段是什么意思
3. 前端按这些稳定字段渲染
4. Job 历史也不再依赖前端本地缓存，而是来自真实后端接口

也就是说，这一步本质上是：

> **把“后端内部快照”升级成“前后端都认可的正式数据契约”。**

---

## 3. 本步为什么选方案 B？
本步最终采用的是：

> **扩报告接口 + 补作业维度 Job 列表接口**

也就是方案 B。

### 3.1 为什么不只改报告接口？
如果只改 `GET /report`：
- 的确能看到统计
- 但 Job 历史列表还是要靠前端 `localStorage` 记住 `jobId`

这会带来两个问题：
- 前端数据来源不统一
- 换浏览器或清缓存后，历史 Job 看不到

### 3.2 为什么不把统计直接塞到实体里？
如果把很多展示逻辑都直接混进 `PlagiarismJob` 实体：
- 后端实体会越来越像“前端专用返回对象”
- 结构会越来越乱
- 后面维护成本高

### 3.3 方案 B 的好处
方案 B 会同时解决两个问题：
- 统计数据正式暴露给前端
- Job 列表也变成真实后端接口，不再靠本地缓存兜底

所以这是“当前阶段最完整、但改动仍然可控”的方案。

---

## 4. 本步的后端设计
### 4.1 扩展报告返回结构
原来的报告接口是：

`GET /api/teacher/plagiarism-jobs/{jobId}/report`

原来主要返回：
- 任务状态
- 筛选参数
- 相似对列表
- 不可比对列表

本步会增加一个新的统计块，例如：

- `jobStats.comparableSubmissionCount`
- `jobStats.comparablePairCount`
- `jobStats.sizeSkippedPairs`
- `jobStats.bucketSkippedPairs`
- `jobStats.fullCalculatedPairs`
- `jobStats.thresholdMatchedPairs`
- `jobStats.executionMode`
- `jobStats.reusedFromJobId`

这样前端拿到报告时，就不用自己解析 `config_snapshot`。

### 4.2 新增作业维度 Job 列表接口
新增接口：

`GET /api/teacher/assignments/{assignmentId}/plagiarism-jobs`

这个接口用于返回某份作业下的 Job 历史列表。  
每条记录会是一个“面向页面展示的摘要对象”，例如：
- Job ID
- 状态
- 创建时间
- 进度
- 阈值
- TopK
- 执行模式（ASYNC / REUSED）
- 来源 Job ID
- 命中相似对数量

这样教师端就能真正展示历史 Job，而不是依赖浏览器缓存。

## 5. 本步的前端设计
### 5.1 作业管理页
教师作业管理页里的“查重与结果”部分，新增两类信息：

#### Job 列表增强
每个 Job 除了状态、进度，还会看到：
- 执行模式
- 来源 Job（如果是复用）
- 命中结果数

#### 报告统计卡
在当前报告摘要区域，增加一块“任务统计”：
- 可比对提交数
- 理论 pair 总数
- 大小上界跳过数
- 粗桶上界跳过数
- 完整 AC 计算数
- 命中阈值数

这能让老师一眼知道：
- 这次任务做了多少工作
- 优化有没有生效

### 5.2 作业详情页
教师查看某个作业详情时，在“查重记录”页签里也展示同样的统计摘要。  
这样老师无论是在总览页还是详情页，都能看到一致的数据。

### 5.3 去掉前端主路径中的 Job 缓存依赖
当前教师端前端的 Job 历史依赖本地缓存 `jobId`。  
本步之后，主数据来源改为真实接口：
- 后端提供 `assignment -> jobs`
- 前端优先使用后端返回
- 本地缓存只保留为兼容兜底，或者直接删除

这样系统行为会更稳定，也更符合真实项目设计。

## 6. 本步如何开发？
### 6.1 先写 DTO，不直接把 JSON 扔给前端
后端不会让前端直接读 `config_snapshot` 原始字符串。  
而是新增专门的 DTO，例如：
- `TeacherPlagiarismJobStatsView`
- `TeacherPlagiarismJobSummaryView`

这样做的好处是：
- 字段名稳定
- 前端更容易对接
- 后续即使内部快照结构微调，也不影响页面

### 6.2 服务层统一解析快照
在 `TeacherPlagiarismJobServiceImpl` 里统一做：
- 解析 `paramsJson`
- 解析 `configSnapshot`
- 组装为前端真正需要的结构

这样 controller 仍然保持很薄，只负责转发。

### 6.3 前端继续保持“接口层 + 映射层”分离
前面教师端前端已经做过一次整理：
- 接口收口在 `teacherAssignments.js`
- 数据映射收口在 `assignmentMappers.js`

本步会继续遵守这个结构：
- API 层只负责请求
- mapper 层只负责转换字段
- Vue 页面只负责渲染

这能保证代码不会越改越乱。

---

## 7. 这一步用了什么技术？
### 7.1 DTO 映射
把数据库实体或快照 JSON 转成专门给前端用的对象。  
这是后端开发里最常见也最推荐的做法，因为它能避免“数据库结构直接外泄到页面”。

### 7.2 快照解析
本步继续复用 `config_snapshot` 思路，但不让前端自己碰原始 JSON。  
而是后端先解析，再输出稳定字段。

### 7.3 前后端契约化
“契约化”听起来很专业，其实意思很简单：

> **前后端提前说好字段名、字段含义、字段类型。**

这一步做的就是把原来“后端内部知道的数据”，变成“前后端都明确认可的接口字段”。

## 8. 这一步完成后会有什么效果？
### 对老师
- 能直接看到这次查重做了多少工作
- 能分辨这是正常执行还是复用历史结果
- 能看到 Job 历史，而不是只看当前一次

### 对开发者
- 不再需要手动翻 `config_snapshot`
- 排查为什么某次任务快、为什么某次任务慢，会容易很多

### 对答辩展示
- 你可以直接展示页面里的统计卡
- 能说明系统不仅能“跑出结果”，还能“解释自己怎么跑出来的”

---

## 9. 一句话总结

**这一步的本质，是把后端已经记录下来的任务统计，正式升级成前后端共享的稳定接口，并接入教师端页面展示，让查重任务不再只是“有结果”，而是“过程也可见、历史也可见、优化效果也可见”。**
