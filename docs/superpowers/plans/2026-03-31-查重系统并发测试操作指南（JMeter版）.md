# 查重系统并发测试操作指南（JMeter版）

## 1. 文档目标

本文档用于指导**不会使用 PowerShell、希望尽量通过图形界面完成测试**的项目成员，从零开始完成一次针对当前查重系统的并发测试。

本次测试的目标不是做复杂的压测平台，而是完成一条**可操作、可复现、可展示**的测试链路：

1. 启动后端与数据库
2. 教师创建测试班级与测试作业
3. 批量准备 100 个学生测试账号
4. 使用 `JMeter` 模拟 100 个学生并发提交作业
5. 教师在前端手工点击“发起查重”
6. 观察 Job 状态与查重结果
7. 到数据库核对结果是否正确

---

## 2. 为什么推荐 JMeter

对于当前项目，`JMeter` 比 `PowerShell` 更适合成果展示和初学者使用：

- 基本通过图形界面操作，不需要写很多命令
- 很适合做“100 个学生同时提交”的并发测试
- 测试流程容易截图，适合答辩展示
- 支持导出统计结果，方便写论文或测试记录

本次建议采用的测试组合如下：

- **JMeter**：模拟学生并发登录与提交
- **浏览器前端**：教师登录、创建作业、手工发起查重
- **Navicat / DataGrip / MySQL Workbench**：查看数据库结果

---

## 3. 本次测试到底测什么

本次测试主要验证两部分能力。

### 3.1 提交阶段并发能力

验证 100 个学生在短时间内集中提交作业时，系统是否能够稳定完成：

- 登录
- 创建 `submission`
- 保存 `submission_file`
- 解析 Java 文件
- 生成 `submission_profile`

### 3.2 查重阶段处理能力

验证教师在 100 份有效提交完成后发起一次查重任务时，系统是否能够稳定完成：

- 创建 `plagiarism_job`
- 执行 `4950` 组代码对比较
- 更新任务进度
- 保存 `similarity_pair`
- 保存 `similarity_evidence`
- 最终生成查重报告

> 说明：100 个学生两两组合的理论对数为：`100 × 99 ÷ 2 = 4950`

---

## 4. 当前系统的并发特点（先理解这个很重要）

### 4.1 学生提交接口可以并发压测

当前用于学生提交文本代码的接口为：

- `POST /api/login`
- `POST /api/student/assignments/{assignmentId}/submissions/text`

其中：

- 登录成功后会返回 `token`
- 后续提交接口需要在请求头中带上 `satoken`

### 4.2 单个查重 Job 是异步执行，但 Job 内部比对是顺序进行的

当前系统配置了异步线程池：

- `core-pool-size = 2`
- `max-pool-size = 4`
- `queue-capacity = 20`

但要注意：

- 教师点击一次“发起查重”，通常只会创建 **1 个 Job**
- 这个 Job 会异步执行
- **单个 Job 内部的 4950 对比较，当前实现是顺序执行，不是并行执行**

所以本次测试要分开理解：

1. **提交并发测试**：测接口抗压能力
2. **查重任务测试**：测后台完成大任务的能力

## 7. 第二步：教师先手工准备测试场景

这一部分建议全部通过前端手工完成，不要自动化。

### 7.1 教师登录前端

使用你的教师账号登录系统。

### 7.2 创建一个测试班级

建议创建一个专门用于压测的班级，例如：

- 班级名称：`并发测试班`

创建后，在数据库中查出这个班级的 `classId`：

```sql
SELECT id, class_name, teacher_id
FROM sys_class
ORDER BY id DESC;
```

记下这个 `classId`，后面要用。

### 7.3 创建一个测试作业

建议教师在前端创建一个专门用于查重测试的作业，例如：

- 作业标题：`查重并发测试作业`
- 允许重复提交：是
- 允许迟交：是
- 最大文件数：5
- 截止时间：设置成未来几分钟或未来几小时

创建后，在数据库中查出这个作业的 `assignmentId`：

```sql
SELECT id, title, teacher_id, clazz_id, start_at, end_at
FROM assignment
ORDER BY id DESC;
```

记下这个 `assignmentId`，后面配置 `JMeter` 时会用到。

## 8. 第三步：批量创建 100 个学生测试账号

这一步建议直接通过数据库批量插入，不要手工注册 100 个学生。

> 说明：当前项目登录逻辑是直接比较明文密码，所以这里示例密码直接使用 `123456`。

### 8.1 批量插入用户表 `users`

```sql
WITH RECURSIVE seq AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1 FROM seq WHERE n < 100
)
INSERT INTO users (username, password, nickname, role, email, status, uid, create_time, update_time)
SELECT
    CONCAT('ptest_stu', LPAD(n, 3, '0')),
    '123456',
    CONCAT('压测学生', LPAD(n, 3, '0')),
    'STUDENT',
    CONCAT('ptest_stu', LPAD(n, 3, '0'), '@test.local'),
    1,
    CONCAT('9', LPAD(n, 9, '0')),
    NOW(),
    NOW()
FROM seq
WHERE NOT EXISTS (
    SELECT 1
    FROM users u
    WHERE u.username = CONCAT('ptest_stu', LPAD(n, 3, '0'))
);
```

检查是否插入成功：

```sql
SELECT COUNT(*) AS cnt
FROM users
WHERE username LIKE 'ptest_stu%';
```

预期结果：`100`

### 8.2 插入学生信息表 `sys_student_info`

```sql
INSERT INTO sys_student_info (user_id, student_number, school, college, major, class_name, create_time, update_time)
SELECT
    u.id,
    CONCAT('2026', LPAD(RIGHT(u.username, 3), 4, '0')),
    '测试学校',
    '计算机学院',
    '计算机科学与技术',
    '压测班',
    NOW(),
    NOW()
FROM users u
WHERE u.username LIKE 'ptest_stu%'
  AND NOT EXISTS (
      SELECT 1
      FROM sys_student_info s
      WHERE s.user_id = u.id
  );
```

### 8.3 把这 100 个学生加入测试班级

把下面 SQL 里的 `1` 替换成你真实的 `classId`：

```sql
SET @classId = 1;

INSERT INTO sys_class_student (class_id, student_id, join_time, status)
SELECT
    @classId,
    u.id,
    NOW(),
    1
FROM users u
WHERE u.username LIKE 'ptest_stu%'
  AND NOT EXISTS (
      SELECT 1
      FROM sys_class_student cs
      WHERE cs.class_id = @classId
        AND cs.student_id = u.id
  );
```

检查是否加入成功：

```sql
SELECT COUNT(*) AS cnt
FROM sys_class_student
WHERE class_id = 1 AND status = 1;
```

## 9. 第四步：准备 JMeter 所需的学生账号 CSV 文件

为了让 `JMeter` 给每个线程分配不同的学生账号，建议准备一个 CSV 文件。

### 9.1 推荐做法：用 Excel 创建 CSV

新建一个 Excel 文件，第一行写表头：

| username | password | studentNo | cluster |
| --- | --- | --- | --- |

从第 2 行开始填数据。

例如前几行：

| username | password | studentNo | cluster |
| --- | --- | --- | --- |
| ptest_stu001 | 123456 | 1 | 1 |
| ptest_stu002 | 123456 | 2 | 2 |
| ptest_stu003 | 123456 | 3 | 3 |
| ptest_stu004 | 123456 | 4 | 4 |
| ptest_stu005 | 123456 | 5 | 0 |

### 9.2 Excel 快速生成方法

假设：

- A 列 = `username`
- B 列 = `password`
- C 列 = `studentNo`
- D 列 = `cluster`

可以使用如下公式：

- `A2`：`="ptest_stu"&TEXT(ROW(A1),"000")`
- `B2`：`="123456"`
- `C2`：`=ROW(A1)`
- `D2`：`=MOD(C2,5)`

然后向下拖到第 101 行，得到 100 个学生。

最后保存为：

- 文件名：`students.csv`
- 编码：建议 `UTF-8`

> 建议把这个文件放到一个你容易找到的位置，例如桌面或项目目录下单独的 `test-data` 文件夹。

### 10.1 安装方式

## 11. 第六步：在 JMeter 中创建测试计划

建议你按下面结构搭建：

```text
Test Plan
└─ Thread Group（学生并发提交）
   ├─ HTTP Request Defaults
   ├─ HTTP Header Manager（登录）
   ├─ CSV Data Set Config
   ├─ HTTP Request（登录）
   ├─ JSON Extractor（提取 token）
   ├─ HTTP Header Manager（提交接口）
   ├─ HTTP Request（文本提交）
   ├─ View Results Tree
   ├─ Summary Report
   └─ Aggregate Report
```

---

## 12. 第七步：配置 Thread Group

右键 `Test Plan`：

- `Add`
- `Threads (Users)`
- `Thread Group`

建议初始配置如下：

| 配置项 | 建议值 | 说明 |
| --- | --- | --- |
| Number of Threads (users) | 5 | 先小规模试跑 |
| Ramp-Up Period (seconds) | 5 | 5 秒内启动完 5 个线程 |
| Loop Count | 1 | 每个学生只提交 1 次 |

**非常重要：**
先不要直接配 `100`，第一次先用 `5`，跑通后再改成 `20`，最后再改成 `100`。

---

## 13. 第八步：配置 HTTP Request Defaults

右键 `Thread Group`：

- `Add`
- `Config Element`
- `HTTP Request Defaults`

填写：

| 字段 | 值 |
| --- | --- |
| Protocol | `http` |
| Server Name or IP | `localhost` |
| Port Number | `8080` |

这样后面的请求就不用每次都填主机和端口了。

---

## 14. 第九步：配置 CSV Data Set Config

右键 `Thread Group`：

- `Add`
- `Config Element`
- `CSV Data Set Config`

填写建议如下：

| 字段 | 值 |
| --- | --- |
| Filename | 你的 `students.csv` 绝对路径 |
| File Encoding | `UTF-8` |
| Variable Names | `username,password,studentNo,cluster` |
| Ignore first line | `True` |
| Delimiter | `,` |
| Recycle on EOF | `False` |
| Stop thread on EOF | `True` |
| Sharing mode | `All threads` |

这样每个线程就会自动拿到一行学生账号数据。

---

## 15. 第十步：配置登录请求

### 15.1 添加登录请求

右键 `Thread Group`：

- `Add`
- `Sampler`
- `HTTP Request`

名称建议：`学生登录`

填写：

| 字段 | 值 |
| --- | --- |
| Method | `POST` |
| Path | `/api/login` |

勾选 `Body Data`，填入：

```json
{
  "username": "${username}",
  "password": "${password}"
}
```

### 15.2 给登录请求添加请求头

右键“学生登录”这个请求：

- `Add`
- `Config Element`
- `HTTP Header Manager`

添加一行：

| Name | Value |
| --- | --- |
| Content-Type | application/json |

---

## 16. 第十一步：从登录响应中提取 token

右键“学生登录”这个请求：

- `Add`
- `Post Processors`
- `JSON Extractor`

填写：

| 字段 | 值 |
| --- | --- |
| Names of created variables | `token` |
| JSON Path Expressions | `$.data.token` |
| Default Values | `TOKEN_NOT_FOUND` |

配置完成后，如果登录成功，`${token}` 变量里就会保存当前学生的登录 token。

---

## 17. 第十二步：配置文本提交请求

### 17.1 添加文本提交请求

右键 `Thread Group`：

- `Add`
- `Sampler`
- `HTTP Request`

名称建议：`学生文本提交`

填写：

| 字段 | 值 |
| --- | --- |
| Method | `POST` |
| Path | `/api/student/assignments/3/submissions/text` |

> 注意：这里的 `3` 要替换成你真实的 `assignmentId`。

### 17.2 提交请求体示例

勾选 `Body Data`，填入：

```json
{
  "entries": [
    {
      "filename": "Main.java",
      "content": "public class Main { public static void main(String[] args) { int x = ${studentNo}; if (x > ${cluster}) { System.out.println(x + ${cluster}); } else { System.out.println(${cluster} - x); } } }"
    },
    {
      "filename": "Helper.java",
      "content": "public class Helper { public static int add(int a, int b) { return a + b + ${cluster}; } }"
    }
  ]
}
```

这个请求体的作用是：

- 每个学生都提交 2 个 `.java` 文件
- 代码结构相似，但会因为 `studentNo` 和 `cluster` 不同而有一点差异
- 适合你后续展示 AST 查重效果

### 17.3 给提交请求添加 Header

右键“学生文本提交”这个请求：

- `Add`
- `Config Element`
- `HTTP Header Manager`

添加两行：

| Name | Value |
| --- | --- |
| Content-Type | application/json |
| satoken | `${token}` |

---

## 18. 第十三步：添加结果查看组件

为了方便你观察结果，建议先添加下面三个监听器。

### 18.1 View Results Tree

右键 `Thread Group`：

- `Add`
- `Listener`
- `View Results Tree`

用途：
- 适合调试
- 可以看每次请求的请求体、响应体、状态码

> 注意：正式压测时它比较占资源，**100 并发正式跑时建议先禁用或删除**。

### 18.2 Summary Report

右键 `Thread Group`：

- `Add`
- `Listener`
- `Summary Report`

用途：
- 看平均响应时间
- 看吞吐量
- 看错误率

### 18.3 Aggregate Report

右键 `Thread Group`：

- `Add`
- `Listener`
- `Aggregate Report`

用途：
- 统计整体成功率和耗时分布

## 21. 第十六步：正式跑 100 个学生并发提交

正式测试建议参数如下：

| 项目 | 建议值 |
| --- | --- |
| Threads | 100 |
| Ramp-Up | 10 ~ 20 |
| Loop Count | 1 |

建议你第一次正式测试用：

- `Threads = 100`

- 20 秒内逐步启动 100 个学生线程

## 23. 第十八步：观察查重任务状态

教师发起查重后，可以通过前端页面观察状态，也可以直接查数据库。

### 23.1 查 Job 状态

```sql
SELECT id, assignment_id, status, progress_total, progress_done, start_time, end_time, error_msg
FROM plagiarism_job
ORDER BY id DESC;
```

你需要重点看：

- `status`：是否从 `QUEUED` 变成 `RUNNING`，最终到 `DONE`
- `progress_total`
- `progress_done`
- `error_msg` 是否为空

### 23.2 理论上应该看到什么

如果当前作业下有 100 份有效的最新提交，则：

- `progress_total` 理论值应接近或等于 `4950`

如果不是 `4950`，一般说明：

- 并不是 100 个有效提交
- 有些提交 `parse_ok_files = 0`
- 有些学生不在该作业班级中
- 有些学生没有最新提交版本

## 24. 第十九步：核对查重结果是否落库

### 24.1 查看相似对数量

```sql
SELECT COUNT(*) AS pair_cnt

WHERE job_id = 你的_job_id;
```

### 24.2 查看证据数量

```sql
SELECT COUNT(*) AS evidence_cnt
FROM similarity_evidence
WHERE pair_id IN (
    SELECT id FROM similarity_pair WHERE job_id = 你的_job_id
);
```

### 24.3 查看提交是否都形成了画像

```sql
SELECT COUNT(*) AS profile_cnt
FROM submission_profile
WHERE submission_id IN (
    SELECT id FROM submission WHERE assignment_id = 3
);
```

### 24.4 查看是否存在不可比对提交

```sql
SELECT id, student_id, parse_ok_files, total_files, is_valid
FROM submission
WHERE assignment_id = 3
ORDER BY id DESC;
```

## 25. 第二十步：如何判断本次测试是否通过

### 25.1 提交阶段通过标准

满足下面几条，可认为提交并发测试基本通过：

- 100 个学生线程中绝大多数成功
- 登录成功率高
- 提交成功率高
- `submission` 成功入库
- `submission_file` 成功入库
- `submission_profile` 成功生成

### 25.2 查重阶段通过标准

满足下面几条，可认为查重任务测试基本通过：

- 教师能成功发起 Job
- Job 状态能从 `QUEUED` 进入 `RUNNING`
- 最终变成 `DONE`
- `progress_done = progress_total`
- `similarity_pair` 和 `similarity_evidence` 成功生成

## 26. 常见问题与排查方法

### 26.1 登录失败

可能原因：

- 测试账号没有插入成功
- 用户名或密码错误
- 用户状态不是启用

检查 SQL：

```sql
SELECT username, password, role, status
FROM users
WHERE username LIKE 'ptest_stu%';
```

### 26.2 学生提交失败

可能原因：

- 学生没有加入测试班级
- `assignmentId` 配错
- 提交请求头里没有带 `satoken`
- `JSON Extractor` 没提取到 `token`

排查建议：

- 先看 `View Results Tree`
- 再看登录响应里是否真的有 `data.token`
- 再看提交请求头中 `satoken` 是否已替换成真实值

### 26.3 解析失败导致不可比对

可能原因：

- 你在 JMeter 里写的 Java 代码格式不正确
- JSON 字符串转义有误

建议：

- 先用 5 人试跑
- 确认 `submission_profile` 正常生成后再上 100 人

### 26.4 Job 卡在 RUNNING 或 FAILED

可能原因：

- 某些数据异常
- 后端执行中报错
- 文件读取或数据库写入异常

检查 SQL：

```sql
SELECT id, status, progress_total, progress_done, error_msg
FROM plagiarism_job
ORDER BY id DESC;
```

并同时查看后端控制台日志。

## 27. 最推荐的测试顺序

为了降低出错概率，强烈建议按下面顺序进行：

1. **5 个学生**：验证 JMeter 配置无误
2. **20 个学生**：验证中等规模可以稳定运行
3. **100 个学生**：作为正式展示测试
4. **教师发起一次查重**：观察 Job 完成情况
5. **数据库核对结果**：确认数据完整性

不要一上来就直接跑 100，否则一旦失败，很难判断到底是：

- JMeter 配置错了
- token 没提取到
- 作业或班级关系有问题
- 提交代码格式不合法
- 后端性能不足

## 30. 结论

对于当前项目，最适合初学者和成果展示的测试方式是：

- 用 **JMeter** 做 100 学生并发提交
- 用 **前端页面** 让教师手工发起查重
- 用 **数据库查询** 验证结果是否完整落库

这套方案的优点是：

- 学习成本低
- 操作可视化
- 容易截图和演示
- 适合毕业设计场景

如果后续需要进一步做更高阶测试，可以在此基础上继续扩展：

- 多个作业同时发起查重
- 更复杂的代码样本分布
- 不同阈值下的结果对比
- 自动生成测试报告
