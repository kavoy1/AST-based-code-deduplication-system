# 查重核心实现评审

## 1. 文档目的

本文档用于对当前后端项目中的查重核心实现进行技术评审，明确当前系统已经实现了什么、主流程是什么、设计是否合理、当前版本的有效性与正确性边界在哪里，以及下一步最值得优化的方向。

本文档面向：

- 后端开发人员
- 前端对接人员
- 项目负责人
- 毕设论文与答辩整理

---

## 2. 当前查重系统的总体定位

当前系统实现的不是传统文本比对型查重，也不是基于树编辑距离（TED）的严格 AST 对齐算法，而是：

- 对 Java 源码进行 AST 解析
- 对 AST 做归一化处理
- 提取一组结构签名（signature）
- 将 signature 统计为 multiset（特征袋）
- 使用 `M = Σ min(count1[s], count2[s])` 计算匹配节点数
- 使用 `AC = 2M / (N1 + N2)` 计算结构相似度

因此，当前系统更准确的表述应为：

**基于 AST 归一化结构特征袋（bag-of-nodes）的代码结构相似度检测系统**。

这一定义与当前需求文档中确认的 AC 口径一致，属于一个适合本科毕业设计实现和说明的版本。

---

## 3. 当前系统已经实现的核心能力

### 3.1 提交阶段能力

当前系统在学生提交作业时，已经实现：

- 支持多个 `.java` 文件提交
- 支持文本提交与文件提交两种入口
- 对提交文件逐个进行 Java 解析
- 成功解析的文件提取 AST 结构特征
- 将多个文件的特征合并为提交级画像
- 记录解析失败文件及失败原因
- 将画像缓存到 `submission_profile`

当前实现采用宽松解析策略：

- 若某提交中有部分文件解析成功，则这些成功文件参与查重
- 若 `parseOkFiles >= 1`，则该提交 `isValid = 1`
- 若 `parseOkFiles = 0`，则该提交不参与两两比对，但会进入不可比对名单

这一策略与之前确定的产品口径一致，适合教学场景。

---

### 3.2 AST 归一化与 signature 提取能力

当前系统在 AST 层面已实现以下处理：

- 标识符名统一归一为 `ID`
- 类型名统一归一为 `TYPE`
- 基本类型统一压缩为 `INT`
- 字符串、数字、布尔、字符、空值等字面量只保留类别
- 针对若干关键结构生成 signature，例如：
  - `BinaryExpr:<op>`
  - `UnaryExpr:<op>`
  - `Assign:<op>`
  - `Call:arity=<n>`
  - `If:else=<0|1>`
  - `For:mask=<...>`
  - `ForEach`
  - `While`
  - `DoWhile`
  - `Return`
  - `Break`
  - `Continue`
  - `Throw`
  - `TryCatch`
  - `Switch`
  - `Literal:NUM / STR / BOOL / CHAR / NULL`

这说明当前系统并不是简单的词法层字符串比较，而是已经进入语法结构层面。

---

### 3.3 查重任务执行能力

教师发起查重后，当前系统已经实现：

- 只取 `is_latest = 1` 且 `is_valid = 1` 的提交
- 只对同一班级内的学生进行两两比较
- 创建 `plagiarism_job`
- 异步执行 Job
- 统计理论可比对对数 `progressTotal`
- 在执行中更新 `progressDone`
- 对超过阈值的 pair 落库为 `similarity_pair`
- 为每个 pair 生成 `similarity_evidence`
- 支持报告查询、分页视图、TopK 视图、状态筛选、导出与 AI 解释

此外，当前系统已经加入了以下工程优化：

- 画像缓存优先读取，避免重复解析源码
- size upper bound 剪枝
- bucket upper bound 剪枝
- pair/evidence 批量写入
- 进度更新节流
- 可复用历史 DONE Job 结果

这说明当前实现已经具备一版完整可运行的查重闭环，而不只是实验性代码。

---

## 4. 当前查重主流程说明

### 4.1 学生提交阶段

学生提交代码后，系统流程为：

1. 创建 `submission`
2. 保存每个源码文件到存储目录
3. 逐文件执行 Java 语法解析
4. 对解析成功文件提取 AST 结构 signature
5. 汇总为提交级画像
6. 将画像写入 `submission_profile`
7. 更新 `parseOkFiles`、`isValid` 等字段

这里需要特别说明：

- 当前系统**不会把完整 AST 树直接落库**
- 当前落库的是**提交级结构画像**，即：
  - `algoVersion`
  - `totalNodes`
  - `signatureCountsJson`
  - `parseFailuresJson`

这种设计可以显著减少存储量和重复解析开销，适合作为 V1 方案。

---

### 4.2 教师发起查重阶段

教师发起查重时，系统执行流程为：

1. 检查是否已过截止时间
2. 查询该作业下所有可比对提交
3. 生成提交指纹 `submissionFingerprint`
4. 检查是否存在可复用的历史 DONE Job
5. 若可复用则直接克隆结果
6. 若不可复用则创建异步 Job 并进入后台执行

这一流程的合理性体现在：

- 避免截止前查重
- 保证 Job 与具体提交快照绑定
- 对相同输入可进行复用，降低重复计算成本

---

### 4.3 Job 内部执行阶段

后台 Job 执行时，系统流程为：

1. 查询当前 assignment 下全部可比对提交
2. 构建每个提交的画像上下文
3. 计算理论可比对对数
4. 仅在同一班级内部进行两两遍历
5. 先做 size upper bound 剪枝
6. 再做 bucket upper bound 剪枝
7. 剪枝通过后再进行完整 AC 计算
8. 若 `AC >= threshold`，则生成 `similarity_pair + similarity_evidence`
9. 周期性更新 Job 进度
10. 完成后将 Job 状态更新为 `DONE`

这一执行流程在工程上是合理的，尤其适合“100 人左右班级”的教学场景。

---

## 5. 当前 AC 算法实现评估

### 5.1 当前实现的优点

当前 AC 算法具备以下优点：

1. **实现简单明确**
   - 公式清楚，行为可解释，便于论文撰写和答辩说明

2. **对变量改名、常量替换较稳健**
   - 由于做了归一化，表层改名难以绕过检测

3. **对轻度结构改写仍有识别能力**
   - 如 if/for/while、运算符、赋值方式等仍会留下结构特征

4. **计算成本较低**
   - 不需要做复杂树对齐或编辑距离计算
   - 适合课堂批量作业查重

5. **证据可解释性较强**
   - 可以输出 topN signature overlap
   - 便于教师理解“为什么系统认为相似”

---

### 5.2 当前实现的局限

当前 AC 算法的根本局限也非常明确：

1. **它不是树结构对齐算法**
   - 当前比较的是 signature 计数，不是完整 AST 结构关系
   - 父子关系、层次深度、语句顺序信息大量丢失

2. **对“同功能但结构显著改写”的区分能力有限**
   - 如果两个程序功能相同，但使用不同控制结构和表达方式，结果会波动较大

3. **对“整体形状”敏感度不足**
   - 同一批 signature 即使嵌套关系不同，也可能得到较高相似度

4. **MethodCall 只保留参数个数，语义损失较大**
   - 不区分调用名，不区分被调对象，不区分库函数语义

5. **所有基本类型统一为 INT，语义进一步压缩**
   - 对语义分析有帮助的信息被主动舍弃

6. **多文件提交只做简单合并**
   - 当前不关心文件边界、类边界、方法边界
   - 更像“整份提交的结构袋”而非“多文件工程结构比较”

所以，当前版本适合说成：

- **结构相似度检测**
- **疑似抄袭辅助发现**

而不适合直接说成：

- **高精度抄袭最终判定系统**

---

## 6. 当前实现是否合理

### 6.1 作为本科毕业设计，当前实现是合理的

如果从本科毕业设计的目标来看，当前实现是合理且完整的，原因如下：

- 有明确的理论定义（AC）
- 有明确的 AST 归一化与 signature 规则
- 有完整的数据流：提交 -> 解析 -> 画像 -> Job -> pair -> evidence -> report
- 有工程优化：缓存、剪枝、异步、批量写入、结果复用
- 有报告、证据、AI 解释、教师处理状态等完整闭环

这一层次已经明显超出“单纯演示算法”的程度。

---

### 6.2 作为真实生产级查重系统，当前实现还不够强

如果以真实生产系统标准衡量，当前版本还有明显不足：

- 相似度模型仍然偏粗粒度
- 缺少人工标注评估数据集支持
- 缺少 Precision / Recall / F1 的实验结论
- 证据高亮和实际打分来源不是完全同一层逻辑
- 单个 Job 内部仍然是串行 pair 计算

因此，当前系统更适合定位为：

**面向教学场景的 AST 结构相似度查重原型系统 / V1 系统**

---

## 7. 当前实现的有效性与正确性评估

这里需要区分两个概念：

- **工程正确性**：系统是否按既定规则正确实现
- **检测有效性**：系统是否真的能较好识别抄袭

### 7.1 工程正确性

从当前代码和已有测试来看，工程正确性是比较好的：

- AC 公式已有单元测试
- AST signature 提取与归一化已有单元测试
- Java 解析成功/失败已有单元测试
- Submission -> profile 缓存逻辑已有单元测试
- Job 执行、缓存读取、批量写入、剪枝、进度控制已有单元测试
- Job 创建、复用、报告过滤已有单元测试

因此可以认为：

**当前系统已经较好地“正确实现了它自己定义的规则”。**

---

### 7.2 检测有效性

从算法特性判断，当前系统对以下情况通常较有效：

- 变量名替换
- 常量替换
- 方法名替换
- 类名替换
- 代码轻度搬运和小幅改写
- 多份作业之间主要控制流保持一致

但对以下情况效果可能下降：

- 对结构做大幅重构
- 将 for 改成 while / stream / 递归 等不同表达风格
- 将单方法拆成多方法、多类协作
- 对整体执行顺序做较多变换
- 使用不同 API 但完成相同功能

因此当前系统的结论更适合作为：

- **疑似相似对发现依据**
- **教师复核辅助工具**

而不是：

- **替代人工的最终裁决工具**

---

## 8. 当前实现中最值得肯定的点

### 8.1 画像缓存设计正确

将 AST 提取结果缓存到 `submission_profile`，避免每次 Job 都从源码重新解析，是当前实现中最有价值的工程设计之一。

### 8.2 Job 与报告体系完整

`plagiarism_job`、`similarity_pair`、`similarity_evidence`、`ai_explanation` 形成了完整的任务与结果链路，适合前后端协同。

### 8.3 剪枝思路实用

当前使用 size upper bound 和 bucket upper bound 两层剪枝，可以显著减少无意义完整计算，符合你的项目规模。

### 8.4 报告侧已经具有教学可解释性

支持不可比对名单、topN evidence、状态标记、教师备注、AI 解释，这些都很适合毕业设计展示。

---

## 9. 当前实现最值得优化的点（按优先级排序）

### 优先级 1：补“准确性验证”而不是盲目改算法

这是目前最重要的优化方向。

当前系统虽然实现完整，但如果没有实验数据支撑，就很难证明“查重有效”。

最应该优先补的是：

- 构建人工标注样本集
- 准备高相似 / 中相似 / 低相似代码对
- 在不同阈值下统计：
  - TP
  - FP
  - FN
  - TN
- 计算：
  - Precision
  - Recall
  - F1

这是让系统从“能跑”走向“可信”的关键一步。

---

### 优先级 2：让相似度更体现“结构形状”

当前系统只看 signature 计数，对整体结构关系不够敏感。

建议下一步引入轻量增强，而不是直接上复杂树编辑距离：

- 在 signature 中加入深度分桶 `depthBucket`
- 或按方法粒度分别建画像，再做聚合
- 或补充少量“父节点 -> 子节点”二元结构 signature

这样可以在不显著增加复杂度的前提下，让结果更像“结构相似”而不是“成分相似”。

---

### 优先级 3：统一“打分依据”和“展示证据”层级

当前打分主要来自 AST signature bag-of-nodes，
但代码高亮比较使用的是行文本规范化重合。

这会带来一个问题：

- 教师看到的高亮证据，并不完全等于系统真正用于打分的依据

建议优化方向：

- 让 evidence 与高亮更紧密关联到 signature 命中
- 至少在界面上明确区分：
  - “结构证据”
  - “文本辅助对照”

这样解释性会更一致。

---

### 优先级 4：细化 signature 规则而不是简单继续压缩

当前规则已经可用，但还有一些信息压缩得过头。

可以考虑逐步增强：

- `TryCatch` 增加 `hasFinally` 或 catch 数量分桶
- `Switch` 增加 `hasDefault` / caseCountBucket
- `Call` 可选引入白名单保留少量标准库调用名
- `BinaryExpr` / `Assign` 的分类继续保持细粒度

注意这里不要大幅增加复杂性，而是做“可解释、可控”的增强。

---

### 优先级 5：提升 Job 内部执行吞吐量

当前 Job 已异步，但单个 Job 内部 pair 计算仍然是串行的。

对你的毕业设计阶段来说，这没有问题；但如果后续班级规模扩大，会成为瓶颈。

后续可考虑：

- 按班级切分 pair 任务
- 按 pair 分块并行
- 将 evidence 生成与 pair 计算拆开

不过这一项优先级低于“准确性验证”和“结构增强”。

---

## 10. 最终结论

### 10.1 当前系统的结论

当前系统已经实现了一套**逻辑完整、工程上可运行、适合毕业设计展示**的 AST 结构相似度查重方案。

它的优势在于：

- 定义清晰
- 流程完整
- 具备缓存、剪枝、异步、报告与证据能力
- 对轻度改名和轻度改写有较好识别能力

### 10.2 当前系统的边界

当前系统的边界也必须明确：

- 它是基于 AST 特征袋的结构相似度检测
- 它不是严格 AST 对齐算法
- 它不能替代教师人工判定
- 它目前更适合作为“疑似相似对发现工具”而不是“自动裁决系统”

### 10.3 最重要的下一步

如果要让系统更有说服力，当前最值得投入的方向不是盲目大改架构，而是：

1. **先做准确性验证实验**
2. **再做轻量结构增强**
3. **最后再考虑性能并行化**

这是最适合当前项目阶段的技术演进顺序。
